# Plan: Notchpad — notch-anchored recording surface

_Locked via grill-with-docs — by Claude + Bruno. Terms per CONTEXT.md. Revised after Codex round 1 (all 16 findings accepted)._

## Goal

Add the Notchpad: an optional recording/status surface fused with the built-in display's camera notch. When enabled in Settings and a valid notched display is awake, every HUD presentation — recording panel (state, elapsed time, streaming transcript preview, lock/Done/Raw/Cancel/language/output/template controls), transient flashes, persistent text, and translation recovery — renders as a panel expanding from under the notch instead of the floating HUD. When no valid notched display is available (clamshell, external-only, unvalidatable geometry), behavior is today's floating HUD — the notch is never emulated or guessed. Sole floating-behavior exception: the restore-base warning correction (step 4), which applies on both surfaces by design. Default off.

## Approach

1. **Setting.** `AppSettings.notchpadEnabled: Bool` (`@Published`, `didSet` → UserDefaults `Keys.notchpadEnabled`, default `false`), following the `edgeLauncherEnabled` pattern. Full Backup Bundle integration: add to `exportableKeys`, init, export snapshot, `SettingsPatch` decode/apply with default `false`, plus backup round-trip/reset tests (per the existing mirrored-schema discipline in `AppSettings.swift` / `BackupBundle.swift`). UI: "Notchpad" section in the Launcher settings tab — `Toggle("Show FreeTalker in the notch")` + caption: all HUD presentations (recording panel, status flashes, translation recovery) move to the notch on the built-in display; falls back to the floating panel in clamshell/external-only setups. Launcher page subtitle updated to cover both the edge launcher and the Notchpad.

2. **Notch geometry, validated.** New `NotchGeometry` value type + a **pure resolver** fed by a main-actor AppKit snapshot adapter (resolver takes captured inputs — `isBuiltIn`, insets, auxiliary rects, screen frame — and never calls display APIs itself). A display qualifies only if (a) captured `CGDisplayIsBuiltin` is true, (b) `safeAreaInsets.top > 0`, and (c) both `auxiliaryTopLeftArea` and `auxiliaryTopRightArea` pass explicit invariants: finite, non-empty, correctly ordered (`left.maxX < right.minX`), top-aligned with and contained in `screen.frame`. Notch width = the gap; **no fallback width** — any invalid/missing geometry ⇒ Notchpad unavailable ⇒ floating HUD (hardware-honest rule). Resolver returns `NotchGeometry` (display ID + frames), not a raw `NSScreen`; selection is deterministic (the built-in display), never `screens.first`. Adversarial geometry tests (empty/inverted/misaligned/out-of-frame/non-finite rects).

3. **One panel, restyled — no second window for content.** Keep the single reusable `HUDPanel` (existing reuse-and-replace-content design). Introduce `SurfaceStyle` (`floating` / `notch`) owned by `HUDController`. **Complete policy per style, floating preserved byte-for-byte:** floating keeps today's exact recipe (level `.floating`, collection `[.canJoinAllSpaces, .fullScreenAuxiliary, .stationary]` — it already has all three — nonactivating, transparent, non-key/main); notch style changes ONLY what differs: level `.statusBar`, positioning, wrapper chrome, drag disabled. Fallback restores the floating policy exactly — zero regression when Notchpad is off/unavailable.

4. **Single presentation path; base + overlay with precise lifecycle.** Refactor `HUDController` so ALL public entry points — `showRecordingPanel`, `flash`, `show(text:)`, `showTranslationRecovery`, `hide` — funnel through one internal `present(mode:lifetime:)`. Lifetimes: `showRecordingPanel`, `showTranslationRecovery`, and `show(text:)` are **persistent base replacements** (matching today: `show(text:)` stays until replaced/hidden). `flash` has two explicit lifetimes: **terminal flash** (default — clears/replaces the base, exactly today's behavior: "Cancelled", post-processing statuses) and **restore-base flash**, selected **explicitly at the call site**: the two in-recording warning calls in `AppCoordinator` change to pass `restoreBase` (the controller cannot infer it — terminal cancellation also flashes while a recording base exists). Implementation starts with an **audit of every `flash` reachable while a recording owns the HUD** (known set so far: the two capture watchdog warnings, and the Voice Edit busy notice "Finish the current recording first" from `handleVoiceEditHotKey`); every member of that set is classified `restoreBase` and the complete set is tested — no other `AppCoordinator` HUD call sites change. Note: restore-base is a **deliberate behavior correction on both surfaces** — today such a warning eventually hides unless another recording render happens; regression tests cover Notchpad enabled AND disabled. **Base updates ≠ overlay replacement — narrowly:** ONLY same-recording `showRecordingPanel` updates (preview ticks, locked-timer seconds) occur beneath a restore-base overlay; EVERY other persistent replacement (`show(text:)`, `showTranslationRecovery`, a new recording's panel, terminal flash) cancels the active overlay, its deadline, and its generation, and presents immediately (preserving today's `show(text:)`-cancels-flash behavior; a terminal flash never obscures a newly started recording). Auto-hide timers carry a **generation token**; a timer only hides its own generation — overlay replacement or explicit hide invalidates it. **Reroute does NOT invalidate the overlay timer:** the single panel remains the timer's target, so the same generation stays valid and the overlay expires on schedule on the new surface. **Timer expiry is a defined transition** — `expire(generation)`: stale generation ⇒ no-op; restore-base expiry ⇒ remove only the matching overlay and present the latest base; terminal expiry ⇒ clear the matching presentation AND atomically mark the controller hidden (no retained state for later routing to resurrect). **`hide()` is atomic:** clears base, overlay, expiry/generation, connector, and marks the controller invisible; while hidden, all routing events (screen, Space, setting) are no-ops.

5. **Routing matrix.** Surface per presentation = `notch` iff `notchpadEnabled` && valid `NotchGeometry` exists; else `floating`. Every `Mode` routes identically (recording panel, flash, text, translation recovery) — one matrix, no per-mode exceptions.

6. **Notch placement that touches nothing it shouldn't.** The interactive window sits **entirely below** `screen.frame.maxY - safeAreaInsets.top` (below the menu-bar strip) — no reliance on per-pixel click-through inside the strip (hit-test `nil` does not guarantee clicks pass through a status-level window to menu extras). Visual fusion with the housing, if kept, uses a **separate narrow connector window** with `ignoresMouseEvents = true` (notch-width, black, non-key, same level) — the one sanctioned second window, because it is provably inert. The connector is a **child lifecycle resource of the notch style**: created/ordered/moved/Space-configured (same collection behavior as the panel), its existence derived from a single invariant — present iff `controllerVisible && surface == .notch` — so it is removed exactly on `hide`, fallback to floating, or display removal, and untouched by flash expiry while the panel stays visible (no remove/recreate churn). It never outlives or lags the content panel; policy tests cover each transition. If the fused look isn't worth the extra window, the panel simply hugs the strip's bottom edge. Content may be wider than the notch (recording row reaches ~460pt) but never overlaps the menu-bar strip or intercepts menu-extra clicks.

7. **Content extraction, drag-free.** The named recording UI today is the private `panelRow` inside `HUDView` (`RecordingPanelState` is a state struct, not a view). Extract shared mode/recording content into a surface-agnostic view; `HUDDragSurface` (which calls `window.performDrag`) is applied ONLY in the floating wrapper. Notch surface: not draggable, no position persistence. Surface-specific shape/material/padding live in the wrappers.

8. **Control parity, concretely — all modes.** The notch surface wires the exact same coordinator closures as the floating panel. Recording panel: Done, Raw, Cancel, `PanelCallbacks.onLock`, language, output, and template callbacks. Text/pill mode: `onPillClick` — a **distinct wiring path** from recording-panel Lock (they route to the same coordinator method today but are separate closures; enumerated and tested separately). Translation recovery: `onRetryTranslation`, `onInsertSourceText`. `.panelFinish` is the recording-state event those callbacks feed — unchanged. Tests assert each notch control invokes the existing coordinator path exactly once and never changes the insertion target.

9. **Display reconfiguration, Space changes, live toggle.** `HUDController` retains enough state (base presentation + overlay + recording base state) to **synchronously re-present** when routing changes. Triggers: `didChangeScreenParametersNotification` (lid close mid-Recording ⇒ immediate handoff to floating surface with identical state; lid open ⇒ immediate reroute to notch), the **existing `activeSpaceDidChangeNotification` observer** (retained: reposition/reassert on the current surface, exactly as today for floating), and a subscription to `$notchpadEnabled` (toggling mid-presentation reroutes the visible presentation immediately). Because it's one panel, a handoff is restyle + reposition, not a rebuild.

10. **Observability, change-triggered only.** Privacy-safe structured logs (os.log) on **state transitions only**: surface changed, geometry validation outcome changed (incl. rejection reason), display ID/built-in status changed, handoff events. Unchanged evaluations (per-tick re-presents, preview renders) log nothing (at most debug-level throttled). Never transcript content. This is the only diagnosis channel from a clamshell dev setup.

11. **Tests (headless-safe, following `FloatingPanelPolicyTests` precedent).** Unit: `NotchGeometry` resolver matrix (built-in flag × insets × auxiliary areas × gap sanity, synthetic values), routing matrix (enabled × geometry × mode), panel policy under `.notch` style (level, nonactivation, collection behavior), placement math (content below safe-area strip, connector bounds), timer generation invalidation, base/overlay lifecycle (terminal vs restore-base flash, base-update-under-overlay, `show(text:)` persistence), settings persistence + Backup Bundle round-trip. Only visual/fullscreen behavior is deferred to hardware smoke.

12. **Hardware acceptance gate (lid open).** Fullscreen support is a gated checklist, not an assumption: native fullscreen apps, auto-hidden menu bar, multiple Spaces, Stage Manager, focus on external display, lid open/close mid-Recording, menu-extra clicks beside the panel, connector click-through, focus retention while operating notch controls, insertion-target preservation after notch interaction. `.statusBar` + `.fullScreenAuxiliary` is the attempt, not the guarantee; failures here block ship or shrink scope (documented), never silently pass.

## Key decisions & tradeoffs

- **Replace, don't coexist:** Notchpad is THE surface on the notched display; floating HUD only as hardware fallback. Avoids duplicate recording UI.
- **Notch always wins over focus:** if a valid notched display is awake, it hosts the surface even when the focused app is on an external display.
- **Hardware-honest, strictly:** no fake notch, no guessed width — invalid geometry means the feature is simply unavailable.
- **One panel, two styles:** reuse the existing `HUDPanel` lifecycle instead of a parallel `NotchPanel`; handoff becomes restyle, not state transplant. (Codex round 1, finding 15.)
- **Full control parity:** hands-free flow must not regress on the notch surface.
- **Routing lives in `HUDController`:** all five entry points funnel through one `present` path. `AppCoordinator` HUD call sites remain untouched except for the four in-recording warning call sites (two watchdog and two Voice Edit guards), which explicitly opt into the `restoreBase` lifetime.
- **Restore-base is a deliberate cross-surface behavior correction** (warnings no longer strand a stale panel / vanish silently), not an accidental regression — tested with Notchpad on and off.
- No ADR: reversible, unsurprising given the glossary entry.

## Risks / open questions

- Fullscreen/Spaces/Stage Manager behavior unverifiable headless — covered by the hardware acceptance gate (12), with structured logs (10) as the remote-diagnosis channel.
- Mid-Recording handoff correctness rests on the retained-presentation refactor (4, 9) — the riskiest refactor; policy tests cover the transitions.
- Bruno's daily setup is clamshell: fallback path is his primary path; notch path needs deliberate lid-open smoke sessions.

## Out of scope

- Hover quick actions on the notch, scratchpad/history in the notch.
- Fake notch / notch emulation on notchless displays.
- Any change to floating HUD behavior when Notchpad is disabled or unavailable — EXCEPT the restore-base warning correction (step 4), a deliberate, regression-tested exception.
- External-display "Dynamic Island" style surfaces.
