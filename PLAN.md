# Plan: Spoken Commands + Language Pin + Recording Panel
_Locked via grill-with-docs — by Claude + Bruno. Terms per CONTEXT.md. Revised after Codex round 1._

## Goal
Three features. (1) **Spoken Commands**: English command phrases in any Dictation are interpreted, not transcribed — structure ("new paragraph", "new line", "bullet point", "numbered list"), wrapping ("quote … unquote", "all caps … end caps"), and correction ("scratch that" removes the most recent sentence/clause). Delivered as built-in Template prompt v4 via the existing upgrade-if-unedited migration. (2) **Language Pin**: persistent Auto/EN/PT toggle in the menu bar forcing Transcript language; optional per-app language rules override the pin. (3) **Recording Panel**: the recording HUD becomes a small button row — cancel, finish+paste, finish+paste raw (verbatim, still recorded to Library), one-shot language choice (beats pin AND app rule, this Dictation only), Template cycling, and lock (during hold).

## Approach

### Feature 1 — Spoken Commands (agent A, small)
1. `Models/Template.swift`: for each built-in, append current v3 prompt to `legacyPrompts[id]` (becomes [v1,v2,v3]) and write v4 prompt = v3 text + a Spoken Commands section: English commands interpreted in any language ("new paragraph" → paragraph break; "new line" → line break; "quote X unquote" → "X" quoted; "bullet point"/"numbered list" → list items; "all caps X end caps" → X uppercased; "scratch that" → delete the most recent sentence/clause before the command). Explicitly instruct: phrases used descriptively ("I added a new paragraph about pricing") are NOT commands — apply only when spoken as an instruction at a natural boundary; when ambiguous, transcribe literally.
2. `SelfCheck.swift`: extend the prompt-upgrade checks — v3 text now in legacyPrompts and upgrades to v4; v4 contains marker substrings ("scratch that", "new paragraph", "unquote") AND the descriptive-use guard sentence; edited templates untouched. Mutation-test. (Scope note, accepted: SelfCheck cannot run an LLM, so command interpretation quality is verified by the manual E2E checklist in step 15, not by an automated corpus.)

### Feature 2 — Language Pin (agent B)
3. `Settings/AppSettings.swift`: `languagePin: String` — "auto" | "en" | "pt", `@Published`, UserDefaults, default "auto". `appLanguageRules: [String: String]` (bundleID → "en"/"pt"), same plist-dict pattern as `appRules`. Both validated on set/load (unknown codes → dropped/auto).
4. Language resolution — pure static `AppCoordinator.resolveLanguage(oneShot: String?, bundleID: String?, appLanguageRules: [String:String], pin: String) -> String?` returning nil for auto-detect or a forced code. Precedence: one-shot (panel) > app rule > pin > auto. Each candidate is normalized INDEPENDENTLY (trim/lowercase, must be "en"/"pt"); an invalid candidate falls through to the next — an invalid one-shot never blocks a valid rule/pin. Resolved at stop time in `stopAndTranscribe` alongside the existing template resolution, using the same already-snapshotted bundleID.
5. Engines: change the protocol — `TranscriptionEngine.transcribe(samples:forcedLanguage:)` (`String?`, nil = auto-detect). WhisperKit's preview path keeps early-cancel: concrete overload `transcribe(samples:forcedLanguage:allowEarlyCancel:)`; the protocol witness calls it with `allowEarlyCancel: false`; live preview calls it directly with `forcedLanguage: nil, allowEarlyCancel: true`. No shared mutable state. Additionally, `CloudSTTEngine`'s error surfacing is hardened while touching it: `badResponse` must carry HTTP status + classified hint ONLY — never the raw response body (same redaction contract as CloudLLMError/ConnectionTest). `stopAndTranscribe` computes the forced language and passes it into `processDictation` as a parameter, so mid-flight pin changes can't retroactively apply. `WhisperKitEngine`: when forced, skip `detectLangauge` and set `options.language` directly. `CloudSTTEngine`: add multipart `language` field only when forced. Recorded-language correctness: when `forcedLanguage != nil`, `TranscriptionOutput.language` is the FORCED code — Cloud STT's `decoded.language ?? "en"` fallback must never record EN for a forced-PT dictation. Known accepted asymmetry (documented in code comment): vocabulary and base-URL are still read inside the engines at transcribe time (pre-existing pattern); language is deliberately snapshot-threaded because the panel one-shot makes it dictation-scoped state. Cloud compat note: `language` is an OpenAI-compatible field; a provider rejecting it surfaces through the existing HTTP status mapping — no special casing beyond the normal error path (README documents the field is only sent when forced).
6. Menu bar (`App.swift` MenuBarContentView): "Language" section after the template ForEach divider — three checkmark Buttons (Auto/English/Portuguese) writing `languagePin`.
7. Settings UI: replace the App Rules row model with a UNIFIED per-app rule row (bundleID → optional template + optional language). A row can be template-only, language-only, or both; the Add flow requires an app plus at least one of the two; REMOVING a row deletes the bundleID from BOTH `appRules` and `appLanguageRules` (no invisible stale language override). Storage stays the two dicts; the row model is a UI-level join over `Set(appRules.keys) ∪ Set(appLanguageRules.keys)`. Stale-template semantics: if a row's template ID no longer exists in the store, the row DISPLAYS "(deleted template)" while its language half (if any) keeps working — resolution already falls back for stale template IDs (`resolveTemplate`) and `resolveLanguage` is independent by design; the row remains user-removable as one unit.
8. `SelfCheck.swift`: resolveLanguage precedence truth table (one-shot beats rule beats pin beats auto; INVALID one-shot falls through to valid rule; invalid rule falls through to pin; nil bundleID skips rules); unified-row join/removal semantics as a pure function (remove clears both dicts). Mutation-test.

### Feature 3 — Recording Panel (agent B, same branch — shares the one-shot language state)
9. `UI/HUDPanel.swift`: recording modes render a horizontal button row inside the existing non-activating panel (mechanism proven for the pill's single tap target; extend to multiple controls). In panel mode the whole-capsule `.onTapGesture` is REMOVED — each control gets its own explicit tap target/callback; no parent gesture exists to double-fire or swallow child taps. Layout contract: fixed `maxWidth: 460` on the panel content; template-name label truncates (`lineLimit(1)`, ~120pt max); locked-mode preview text is the first thing dropped when width is tight; EN/PT/lock/cancel/done/raw are fixed-size and never dropped. Buttons:
   - ✕ Cancel → existing `cancelRecording` path (same as Esc).
   - ✓ Done → existing stop+transcribe (pill-click semantics: InsertionTarget snapshotted BEFORE the click's state transition, reusing `handlePillClick`'s pre-snapshot pattern for ALL panel buttons that stop).
   - "Raw" → stop+transcribe with skipPostProcessing (step 11).
   - EN | PT — one-shot language buttons: tap sets/clears `oneShotLanguage` for the in-flight Dictation (highlight active choice; tap again clears back to standing behavior). Cleared automatically at every recording start and after every stop/cancel.
   - Template cycle button showing the Active Template's name → advances to the next template in store order. This CHANGES THE GLOBAL Active Template (CONTEXT.md updated to say so) — not a one-shot; stop-time read makes it apply naturally. NO popup menus on the non-activating panel.
   - Lock 🔒 — visible only in pttRecording state (locked state already shows the lock glyph + timer); routes the existing pillClick state-machine event.
   Layout: per the contract above; elapsed/cap timer remains, preview text yields first (locked mode).
10. `RecordingStateMachine`: Done/Raw CANNOT reuse `pillClick` — from `pttRecording`, pillClick enters locked mode, but Done/Raw must stop immediately. New event `.panelFinish` → `.stopAndTranscribe` from BOTH `pttRecording` and `locked` (and `.none` from idle). The raw flag rides OUTSIDE the state machine (pipeline config, not lifecycle). Lock stays the only control mapped to `pillClick`; Cancel maps to the esc/cancel event. Stale-click guard: EVERY panel callback first routes through the state machine from the CURRENT state and no-ops when the state is no longer a recording state — a second click on Done/Raw/Cancel (or two near-simultaneous controls) after the first already transitioned to idle/processing must do nothing. This decision is part of the pure transition table (idle + panel events → .none), selfcheck-covered.
11. Raw path: `stopAndTranscribe(skipPostProcessing:)` threads to `processDictation` — when set, `refined = transcript` (Post-Processor never invoked) and the Library row records template name `"Raw Transcript"` (a reserved display name, NOT a stored Template) — this both honors the Library's "Template = instructions applied" meaning and makes raw rows distinguishable from post-processing-fallback rows (which keep their real template name). The name is RESERVED: `TemplateStore.upsert` rejects (case-insensitive, trimmed) creating/renaming a Template to "Raw Transcript", with inline UI feedback; AND at store load, any pre-existing user template already named "Raw Transcript" (from before this release) is renamed to "Raw Transcript (Template)" and persisted — the sentinel is unambiguous from first launch. No schema change. HUD flash wording distinguishes "Pasted (raw)".
12. `AppCoordinator`: `oneShotLanguage: String?` — cleared via `defer` in EVERY terminal path of `stopAndTranscribe` (including the empty-samples early return) and in `cancelRecording`, and reset at `beginCapture`. Consumed into the forced-language snapshot before the pipeline Task starts. Panel button callbacks wired like onPillClick; all stop-triggering buttons snapshot InsertionTarget before side effects.
13. `SelfCheck.swift`: panel action routing (each button → expected coordinator action, pure decision table); one-shot lifecycle (set → consumed at stop resolution → cleared; cleared on cancel; cleared on new recording); raw path (processor not invoked, refined == transcript, Library row written). Mutation-test.

### Integration
14. Agents A and B run in parallel worktrees off the same base; expected conflict only in `SelfCheck.swift` (both append). Merge, dedupe, `make app` + `make selfcheck` green, README: Spoken Commands list, Language section, Recording Panel description.
15. Manual smoke-test checklist (Bruno, post-merge — the riskiest behavior is UI event delivery on the non-activating panel and LLM command interpretation, neither reachable by SelfCheck):
   - Panel: each button clickable during hold AND locked mode without focus leaving the target app; Done pastes into the app that was focused when Done was clicked; Raw pastes verbatim; Cancel discards; no double-fire (one click = one action); EN/PT highlight toggles; template cycle updates label and menu-bar checkmark.
   - Language: pin PT → EN dictation forced PT; WhatsApp rule PT + pin EN → PT in WhatsApp; panel EN tap overrides both once, next dictation reverts.
   - Spoken commands (E2E, LLM-dependent): "new paragraph", "quote hello unquote", "scratch that" behave; descriptive "I added a new paragraph about pricing" is NOT treated as a command. Weak-model caveat: Apple FM fallback may under-apply commands — acceptable, correct by editing template or using cloud.

## Key decisions & tradeoffs
- **EN-only command phrases** (work in PT dictations too) — one command vocabulary, no per-language phrase tables; PT phrases can come later as a v5 prompt.
- **"Scratch that" = last sentence/clause, LLM-judged** — matches natural correction; v3 self-correction collapse already covers most rewording.
- **Pin is persistent; app rule > pin; panel one-shot > both** — standing intent lives in pin/rules, in-the-moment intent on the panel; the panel choice can never be silently overridden.
- **One-shot resolved at stop time, passed as parameter through the pipeline** — no mid-transcribe AppSettings reads for language; mid-flight setting changes can't retroactively alter an in-flight Dictation.
- **Raw Dictations still recorded to Library** — keeps the "Library = all past Dictations" contract; Redo Last and re-process work on them.
- **Template switching on the panel = cycle button, not a menu** — popup menus risk activating/focus-stealing on a nonactivating panel; cycling needs only tap targets. Menu bar remains the full picker (and already applies mid-recording via stop-time read).
- **Prompt v4 via existing upgrade-if-unedited migration** — edited templates keep user text; only pristine built-ins upgrade.

## Risks / open questions
- Multiple SwiftUI Buttons on the nonactivating NSPanel: proven for one tap target only. Agent B verifies via build + code reasoning; fallback = discrete onTapGesture zones (same hit-testing path as today's pill).
- Command false positives (descriptive "new paragraph" phrases) — prompt guards against it; residual risk accepted, correctable by editing the template.
- Panel width on small screens: cap + wrap or drop preview text if needed.

## Out of scope
- PT command phrases (future v5); custom user-defined commands.
- Pause/resume; mic switching mid-recording; popup menus on the panel.
- Streaming insertion (separate future project).
- Cloud STT language *detection* changes beyond the forced-language field.
