# Plan: Ollama provider preset + stronger Disfluency/Self-correction cleanup
_Locked via grill-with-docs ‚Äî by Claude + Bruno. Terms per CONTEXT.md. Revised after Codex round 1._

## Goal

The Post-Processor pipeline (Transcript ‚Üí Template ‚Üí Refined Output, local-first, BYOK cloud optional) already exists end-to-end. This iteration closes three gaps: (1) make Ollama's cloud API (user has Ollama Pro) a first-class BYOK provider choice, (2) strengthen the four Built-in Templates so Disfluencies ("hmm", "ahnnn", accidental repetitions) are removed and Self-corrections ("I'll do A‚Ä¶ actually B" ‚Üí "I'll do B") collapse to final intent, (3) migrate the improved prompts into an already-persisted `templates.json` without clobbering user edits, then verify the cloud path end-to-end with a real key.

## Approach

1. **Ollama provider case.** Add `case ollama` to `LLMProviderKind` (`Settings/AppSettings.swift:8`). In `CloudLLMProcessor.process` (`Engines/CloudLLMProcessor.swift`), route `.ollama` through the existing `callOpenAICompatible`. Endpoint verified empirically 2026-07-09: unauthenticated `POST https://ollama.com/v1/chat/completions` ‚Üí 401 (exists, auth-gated; a missing route would 404). Contingency: if E2E reveals payload incompatibility, implement a native `/api/chat` call+decoder in the same file ‚Äî but only then.
2. **Provider-aware defaults.** Known-defaults table per provider: anthropic ‚Üí current Anthropic base URL/model defaults (source: `AppSettings.swift:238-239`); ollama ‚Üí `https://ollama.com/v1` / `gpt-oss:120b`; openAICompatible ‚Üí no known default. Pure helper `resolveProviderDefaults(provider:baseURL:model:)`: if base URL (resp. model) is empty or verbatim-equals ANY known provider default other than the current provider's, replace it with the current provider's default ‚Äî or clear it when the current provider has none (openAICompatible). A user-customized value (matches no known default) is never touched. All matching ("empty", "equals a known default") operates on whitespace-trimmed values, and the request snapshot uses the trimmed strings. Applied both on `llmProvider` `didSet` and once in `AppSettings.init` after loading persisted values (didSet doesn't fire during init; helper is idempotent). `CloudLLMProcessor` validates that BOTH trimmed base URL and trimmed model are non-empty before any request (all three providers require a model); either missing ‚Üí typed error through the shared fallback reporter.
3. **Provider-scoped API keys (security).** Replace single `Keychain.Account.cloudLLMKey` usage with per-provider accounts (`cloudLLMKey-anthropic`, `cloudLLMKey-openAICompatible`, `cloudLLMKey-ollama`). One-time migration, idempotent and checked: skip entirely if the current provider's account already has a value; otherwise copy legacy `cloudLLMKey` value to the current provider's account, verify the write succeeded (read-back or status), and only then delete the legacy item ‚Äî a failed write leaves the legacy key intact. Migration logic written over a minimal injectable key-store interface (get/set/delete) so SelfCheck exercises it with an in-memory fake; the real Keychain adapter stays thin. `CloudLLMProcessor` and the Settings `SecureField` read/write the active provider's account, so switching providers can never send one provider's secret to another's endpoint.
4. **Settings snapshot.** Superseded by Amendment A1: `AppCoordinator` captures ONE snapshot (provider, base URL, model, key, vocabulary) at processor-selection time and passes it into `CloudLLMProcessor` ‚Äî the processor never re-reads settings or Keychain mid-flight.
5. **Settings UI.** Provider Picker (`UI/SettingsView.swift:147-150`) gains `Text("Ollama").tag(LLMProviderKind.ollama)`. Base URL/Model fields show the active provider's defaults via placeholder when applicable. SecureField binds to the active provider's Keychain account and reloads on provider change.
6. **Strengthened Built-in Template prompts.** _Post-E2E note: the strings below are the v2 prompts as approved; E2E showed cross-sentence Self-corrections survived them, so a v3 revision (adds "corrects or contradicts something they said earlier ‚Äî even in a previous sentence ‚Ä¶ drop the superseded statement entirely") superseded them during build. The v2 strings moved verbatim into `legacyPrompts` per step 7's own mechanism; v3 texts live in `Models/Template.swift` and were E2E-verified by Bruno._ Replace the four prompt strings in `Models/Template.swift` with exactly these:
   - **Clean Dictation** (`clean-dictation`): "Clean up this raw speech transcript. Fix punctuation and capitalization. Remove accidental disfluencies: filler words and hesitations (um, uh, hmm, like, you know, and their equivalents in the transcript's language), stutters, accidental word repetitions, and false starts. When the speaker corrects themselves mid-thought (e.g. \"I'll do A‚Ä¶ actually, I'll do B\"), keep only the final version. Preserve intentional repetition, quoted text, names, and emphasis. Do not change the meaning, tone, or wording otherwise. Output only the cleaned text, no commentary."
   - **Refined Message** (`refined-message`): "Rewrite this raw speech transcript as a clear, well-structured chat message, in the same language as the transcript. Remove filler words, hesitations, stutters, and accidental repetitions. When the speaker corrects or revises themselves mid-thought, keep only their final intent. Preserve intentional repetition, quoted text, and names. Fix grammar and organize the ideas so they read as one coherent message. Keep the original meaning and tone. Output only the rewritten message, no commentary."
   - **Refined Prompt** (`refined-prompt`): "Rewrite this raw speech transcript as a clear, precise prompt suitable for sending to an AI assistant, in the same language as the transcript. Remove filler words, hesitations, and false starts. When the speaker revises themselves mid-thought, keep only the final intent. Make the intent explicit and organize multi-part requests into a short list if helpful. Preserve quoted text and names. Output only the rewritten prompt, no commentary."
   - **Email** (`email`): "Rewrite this raw speech transcript as a professional email body, in the same language as the transcript. Remove filler words, hesitations, and accidental repetitions; when the speaker revises themselves, keep only the final intent. Fix grammar and add appropriate structure (greeting/body/sign-off only if implied by content). Preserve names and quoted text. Output only the email body, no commentary."
7. **Upgrade-if-unedited migration (pure function).** Superseded prompts kept verbatim in `Template.legacyPrompts: [String: [String]]` (built-in ID ‚Üí prior defaults). New pure static `Template.upgradingBuiltIns(_ templates: [Template]) -> (templates: [Template], changed: Bool)`: for each element whose `id` is a built-in ID and whose whitespace-trimmed prompt matches a legacy default, replace prompt with the current built-in prompt; `name` untouched (`useCloud`: superseded by Amendment A ‚Äî field removed). Idempotent. `TemplateStore` init applies it after decode and calls `save()` only when it seeded built-ins or `changed == true` ‚Äî no rewrite of unchanged files. **Deliberate: built-ins deleted by the user are NOT re-seeded** ‚Äî deletion is a user edit; migration respects it (Codex finding 8 rejected with this rationale).
8. **Fallback visibility.** One shared fallback reporter, used by BOTH silent-fallback sites ‚Äî `processDictation` and `reprocess` (`AppCoordinator.swift:496`) ‚Äî and for BOTH fallback causes: thrown post-processing errors AND empty/whitespace-only processor output. Non-blocking HUD notice ("Post-processing failed ‚Äî used raw transcript", existing HUD flash mechanism) + log of provider label and HTTP status only. `CloudLLMError.badResponse` drops the response-body payload from its `LocalizedError` description (status code + provider label only); bodies are never logged or surfaced. Fallback behavior itself unchanged ‚Äî never lose the user's words.
9. **SelfCheck additions** (runnable verification; CLT-only env, no XCTest): (a) unedited legacy prompt upgrades; (b) edited prompt untouched; (c) `upgradingBuiltIns` idempotent (second pass ‚Üí `changed == false`); (d) deleted built-in not re-seeded; (e) `resolveProviderDefaults`: empty ‚Üí provider default, other-provider default ‚Üí provider default (or cleared for openAICompatible), custom value ‚Üí untouched, idempotent; (f) Keychain-key migration over in-memory fake store: migrates once, idempotent, failed write preserves legacy key; (g) existing checks still pass. All against pure functions / injected fakes ‚Äî no file or real-Keychain IO in checks.
10. **Build + verify.** `make app && make selfcheck` ‚Äî output shown before claiming done.
11. **E2E manual test (user-driven, guided).** Bruno pastes his Ollama key in Settings (stored in `cloudLLMKey-ollama`), selects Ollama provider (URL/model auto-default), confirms cloud config is complete (Amendment A: no per-Template toggle), dictates a deliberately rambling sentence with fillers and a Self-correction, and we check the Refined Output: fillers gone, only final intent kept; a forced misconfiguration shows the new HUD fallback notice.

## Key decisions & tradeoffs

- **Ollama = enum case reusing OpenAI-compatible wire format** ‚Äî verified live (401 ‚âÝ 404 probe), native `/api/chat` only as contingency. Additive enum case ‚Üí persisted rawValues unaffected.
- **Provider-aware defaults via known-defaults equality** instead of per-provider stored settings: no schema change; only values the user never customized are swapped.
- **Provider-scoped Keychain accounts**: prevents cross-provider secret leakage (Anthropic key sent to ollama.com or vice versa) ‚Äî security, not polish. Legacy account migrated once then removed.
- **Upgrade-if-unedited over version field**: prompt-equality against known legacy strings needs no schema change. Invariant, precisely: semantic edits are preserved; boundary-whitespace-only edits may be upgraded (trim-whitespace comparison is deliberate ‚Äî guards serialization drift, and whitespace-only edits carry no intent worth protecting). Trade-off: each future prompt revision appends its predecessor to `legacyPrompts`.
- **Deleted built-ins stay deleted** ‚Äî user intent respected over "ships with four built-ins" invariant; CONTEXT.md's Built-in Template term already scopes upgrades to never-edited ones.
- **Cleanup rules live in Template prompts, not `buildProcessorInstructions`**: custom Templates opt out by omission; core instruction assembly stays behavior-neutral.
- ~~Built-ins stay `useCloud: false`~~ ‚Äî superseded by Amendment A (field removed; cloud selection is global-implicit).

## Risks / open questions

- `legacyPrompts` must cover every historical default (git history is the source); a missed one degrades to "no upgrade", never breaks.
- Ollama model naming under Bruno's Pro plan confirmed at E2E; 4xx surfaces via `badResponse` ‚Üí HUD notice + raw-Transcript fallback, no data loss.
- Apple FM (local) may follow strengthened prompts less reliably than cloud models; acceptable ‚Äî same fallback contract.

## Out of scope

Streaming LLM responses; per-Template provider/model selection; local Ollama (`localhost:11434`) preset and no-key handling; retry/queue logic; changes to STT engines or Insertion.

# Amendment A: implicit cloud selection (replaces per-Template `useCloud`)
_Added post-E2E by Claude + Bruno; E2E showed the per-Template toggle is a footgun (3 failed rounds traced to it being off)._

## Approach

A1. **Selection rule.** `AppCoordinator` picks the Post-Processor globally: if the active provider's cloud config is complete (trimmed base URL non-empty, trimmed model non-empty, provider-scoped Keychain key present and non-empty) ‚Üí `CloudLLMProcessor`; else `AppleFMProcessor`. Pure helper `isCloudLLMConfigured(snapshot:)` ‚Äî selfcheck-testable. Applies to `processDictation` and `reprocess` alike. ONE snapshot (provider, base URL, model, key, vocabulary) is captured at selection time and passed into `CloudLLMProcessor` ‚Äî routing decision and request always agree; the processor no longer re-reads Keychain/settings mid-flight (Codex round-5 finding 4).
A2. **Remove `Template.useCloud`.** Drop the field from the struct, built-ins, template editor UI, and the `template.useCloud ?` branch. Persisted `templates.json` with the old key still decodes (unknown keys ignored); `upgradingBuiltIns` equality untouched (compares id + prompt only). Custom decoding default not needed ‚Äî field is gone entirely.
A3. **Settings caption + recovery hint.** Cloud post-processing section caption: "Used for all templates whenever provider, model, and API key are configured. Clear the key to go back to on-device processing." Fallback HUD text gains the recovery hint: "Cloud post-processing failed ‚Äî used raw transcript (check API key/model in Settings)".
A4. **Failure contract unchanged.** Cloud failure ‚Üí shared fallback reporter ‚Üí raw Transcript. No FM second attempt.
A5. **SelfCheck**: `isCloudLLMConfigured` truth table (missing key / blank model / blank URL / complete); template decode of legacy JSON containing `useCloud: false` and assertion that routing under complete config is cloud (documented intended behavior ‚Äî user-requested implicit semantics); existing checks green.

## Out of scope (unchanged)
Global manual override toggle; per-Template provider/model; FM-then-raw double fallback.

# Amendment B: hands-free recording (tap = toggle, hold = PTT)
_Locked via grill by Claude + Bruno (gesture model chosen from three options)._

## Approach

B1. **Gesture model.** On hotkey key-down while idle: start capture immediately (unchanged). On key-up: if elapsed since key-down < TAP_THRESHOLD ‚Üí enter LOCKED mode (recording continues); else (held longer) stop + transcribe (classic PTT, unchanged). While LOCKED: next hotkey key-down ‚Üí stop + transcribe; that press's key-up is ignored (`locked + keyUp ‚Üí no-op` is an explicit transition). Esc while recording (either mode) ‚Üí cancel. The event tap detects Esc (keyCode 53) ONLY while recording and swallows it then; when idle, Esc passes through untouched ‚Äî the swallow/pass decision is a pure function in SelfCheck. TAP_THRESHOLD is a named constant, default 0.4 s.
B1a. **Cancel semantics.** One terminal `cancelRecording` action, used by Esc from either mode: stop audio capture, cancel live preview, invalidate the cap timer, clear/flash HUD, discard samples; no transcription, no Library entry, no failed-audio save. SelfCheck asserts the side-effect set via injected hooks.
B2. **Duration cap.** LOCKED recordings auto-stop + transcribe at a configurable cap ‚Äî `AppSettings.handsFreeMaxMinutes`, default 5, clamped to 1‚Äì60 on both read and persist (clamp selfcheck-tested), editable in Settings (General). The clamped cap is snapshotted when entering LOCKED and tied to that recording's generation ‚Äî mid-recording Settings changes affect only future recordings. PTT (held) recordings remain unbounded as today. Double-stop protection: each recording carries a generation ID; every terminal transition (stop, cancel, cap) invalidates the cap timer, and a stale `capReached(gen)` for a finished generation is a no-op ‚Äî covered in the SelfCheck transition table.
B3. **HUD.** While LOCKED the pill shows a distinct state (lock glyph + elapsed time). Clicking the pill: locks an in-progress PTT recording (entering `locked(ignoreNextKeyUp)` so the inevitable physical key release is a no-op), or stops+transcribes a LOCKED one. The current HUD sets `ignoresMouseEvents = true` ‚Äî the pill becomes an `NSPanel` subclass overriding `canBecomeKey`/`canBecomeMain` ‚Üí `false` with `.nonactivatingPanel`, and mouse events enabled deliberately for the pill only. On a click-stop, the insertion target is snapshotted BEFORE the click is processed, so the click itself can never redirect the paste. HUD rendering is mode-aware: in LOCKED mode the layout is lock glyph + elapsed/cap, with live-preview text embedded INSIDE that layout ‚Äî preview ticks update the embedded text only and can never replace the lock state.
B4. **Insertion target.** PTT path keeps its key-up snapshot (unchanged). LOCKED path snapshots the frontmost app/element at STOP time (tap/click/cap), not at the initial key-up ‚Äî the user may have switched apps while speaking; Refined Output pastes where they are when they stop.
B5. **Pipeline unchanged.** Same runPipeline/processDictation contract, template resolution at stop time for LOCKED (same reason as B4), live preview continues working in both modes.
B6. **State machine, pure + testable.** Recording state (idle / pttRecording / locked(ignoreNextKeyUp:Bool)) transitions extracted as a pure function over events (keyDown, keyUp(elapsed), pillClick, esc, capReached(gen)) ‚Äî SelfCheck drives the full transition table, including: tap‚Üílocked, locked+keyDown‚Üístop, locked+keyUp‚Üíno-op, pttRecording+pillClick‚Üílocked(ignoreNextKeyUp) then keyUp‚Üíno-op, held-past-threshold keyUp‚Üístop, esc from both modes ‚Üí cancel, capReached only terminal in locked and only for the live generation.

## Out of scope (B)
Double-press gestures; second lock key; auto-stop on silence; changes to STT engines.
