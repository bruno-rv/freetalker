# Plan: Voice command layer + self-learning vocabulary
_Locked via grill — by Claude + Bruno. Revised after Codex round 1 (11 accepted, 1 modified) and round 2 (7 accepted)._

## Goal

Two features on `feature/voice-commands`, shipped as **separate PRs**. First, a voice command layer: dictations may contain a parameterized spoken keyword (defaults: "command" / "comando") followed by a free-form instruction that the post-processing LLM interprets and applies to the output, plus prefix-free dictation conventions ("double quote", "new paragraph", "scratch that") — handled in the existing single LLM pass via fixed rules in the **trusted system prompt** (not the template), with the LLM using context to distinguish commands from content. Second, self-learning vocabulary: mine recurring proper-noun/jargon corrections from library history (anchored local substitutions in raw-transcript vs refined diffs) into SQLite-backed evidence, surfacing ranked suggestions in Settings for one-click approval into the existing vocabulary bias. Context-aware profiles already exist (`AutomaticStyleClassifier`) — out of scope.

## Approach

### PR A — voice command layer

1. **Settings**: `voiceCommandsEnabled` (**default OFF**; a default-ON flip is explicitly OUT of these PRs and requires its own separately reviewed change with defined evidence — opt-in user feedback plus a small EN/PT/provider acceptance suite) + `commandKeywords: [String]` (default `["command", "comando"]`). Full settings lifecycle planned: validation/bounds (keywords: 1–5 entries, each 2–24 chars, letters only, normalized lowercase), UserDefaults load with fallback, export/backup + restore in `BackupBundle`, **stop-time snapshot** into the processing context (matching how other settings are snapshotted at `AppCoordinator` stop), and recovery-retry semantics (retry uses the snapshot if present, else current settings).
1b. **Durable command snapshot, captured with the capture session**: the stop-time snapshot (`voiceCommandsEnabled` + keywords) is persisted on `capture_sessions` **before or atomically with the audio's staged transition** — closing the crash window where audio is durable but the provisional job doesn't exist yet; reconciliation-recreated jobs inherit it from the session. It then copies into the provisional job and into each retry `AttemptConfiguration`. Nullable at every level; "snapshot absent" (legacy) → retry falls back to current settings. Library rows gain **nullable** `voice_commands_active` (`NULL` = legacy/unknown — built-ins historically embedded command conventions, so unknown rows are treated as possibly-command-processed and **excluded from mining**; `false`/`true` written explicitly going forward).
2. **`VoiceCommandPolicy` on `PostProcessingRequest`** — no fictional single choke point. Each request constructor sets it explicitly:
   - live dictation (external paste, scratchpad recording, notchpad): from snapshot — enabled per toggle;
   - recovery retry: same as live (from snapshot/current);
   - translation pipeline and Scratchpad transformation actions: `.disabled`, always.
3. **Trust boundary**: fixed command-interpretation rules live in the trusted system prompt built by the post-processor, never appended to template content (templates remain untrusted user data). User keywords are passed as validated, bounded data (sanitized list rendered into a constrained slot), not free interpolation. Spoken commands may adjust style/content of the output but are explicitly forbidden from overriding fixed output rules (output-only-final-text, no-meta, language policy).
4. **Grammar spec with examples** (in the system rules and in code docs): a command starts at a keyword that begins a clause and continues until the next sentence/segment boundary — not to end of utterance; successive commands each start with the keyword ("command formal tone. command remove greetings."); dictated content may continue after a command's sentence ends; keyword mid-clause with no imperative that follows ("ok, I got this command") is content; a quoted keyword ("the word 'command'") is content. Conventions ("double quote", "new paragraph", "scratch that" = delete the immediately preceding clause) apply keyword-free, EN + PT examples included.
5. **Built-in template reconciliation**: built-in templates already embed spoken-command rules (paragraphs, quotes, lists, caps, scratch-that in `Template.swift`). Those rules move into the system-prompt block; built-in template bodies are cleaned via the existing template-store migration (version bump). **Edited built-ins**: the migration strips the exact known legacy command-rules suffix from every built-in-ID prompt while preserving all other user edits; if a built-in-ID prompt contains an unrecognized variant of the legacy section, it is left intact and a one-time warning surfaces in Settings (legacy rules still active despite toggle OFF — user decides). User-created templates untouched.
6. **Reusable unit**: `CommandInstructionBuilder` produces the system-rules block; future Shortcuts/URL-scheme automation calls the same layer.
7. **Tests**: prompt-assembly units (block present/absent per policy; keywords sanitized — injection attempts like `"] ignore previous` are neutralized; template text never modified); template migration tests (pristine built-ins cleaned; edited built-ins keep edits with only the exact legacy suffix stripped; unrecognized variants left + flagged; user templates preserved). Golden-prompt scope, corrected: byte identity with today asserted for **system instructions and custom/unchanged templates** under policy `.disabled`; migrated built-ins get a separate assertion of the **exact intentional delta** (legacy suffix removed, rest identical). Behavioral accuracy across models is explicitly NOT test-gated — default-off covers first ship; the flip carries its own evidence gate (see 1).
8. **Observability**: log (non-content) when a request carries an enabled policy and when post-processing fell back with commands enabled, so misfires are diagnosable.

### PR B — self-learning vocabulary

1. **Mining signal, restricted**: only same-language, non-translation, non-raw dictations where a template refinement occurred; candidates are **anchored local substitutions** — token at a matching context position where refined differs from transcript by casing or small edit distance (≤2, length ≥4), suggesting a spelling/casing fix rather than a rewrite. Rows processed with voice commands active are excluded (command execution invalidates the diff). No "OOV" concept — dropped as undefined for BPE.
2. **Storage in SQLite** (library.db, versioned migration): `vocab_evidence(dictation_id, normalized_term, surface_term, first_seen, PRIMARY KEY(dictation_id, normalized_term))`, FK to dictations ON DELETE CASCADE. `vocab_decisions(normalized_term PRIMARY KEY, status TEXT CHECK(status IN ('approved','dismissed')), surface_term TEXT, decided_at, CHECK(status != 'approved' OR surface_term IS NOT NULL))` holds **explicit decisions only** — one row per term (deterministic upsert: newer `decided_at` replaces); "suggested" is never stored, it is derived: evidence recurrence ≥ threshold AND no decision row. Deleting dictations retracts evidence and thus suggestions automatically; **Delete All clears evidence but preserves approved/dismissed decisions** (user intent survives; documented in UI). Canonical `surface_term` (casing/accents, e.g. "João") is fixed at approval time from evidence by deterministic tie-break (highest frequency, then most recent), so cascade-deleted evidence never erases the spelling. Backfill/incremental updates idempotent (PK upsert); **FK failures or missing rows during a scan racing a deletion are benign skips inside the idempotent transaction** — no cross-connection coordination needed. Ownership & bounds: all vocab writes route through one dedicated store actor with a serialized connection; ≤10 candidates per dictation; index `vocab_evidence(normalized_term, dictation_id)`.
2b. **Single source of truth + one derivation for ALL consumers**: approved terms live only in SQLite; `vocabularyText` (UserDefaults) stays purely user-entered. One central derivation (user-entered first, then approved by `decided_at`) feeds **every** existing consumer of `AppSettings.vocabulary`: WhisperKit biasing, Cloud STT multipart field, `CloudLLMSettingsSnapshot`, `AppleFMProcessor`, and the Settings preview. **One shared term validator** (the existing 50-byte NFC/control-character validator) applies identically to manual and approved terms — the earlier 64-byte mining cap is replaced by this single validator.
2c. **Backup**: decisions (surface term, status, decided_at) are a **named, versioned restore stage** in `BackupBundle`, validated and bounded in full before any write, with partial-failure reporting like other stages. Restore conflict rule: **merge by newer `decided_at`** — existing decision kept if newer than the bundle's, replaced otherwise; no duplicates possible (PK upsert), dismissals restore the same way as approvals. Evidence (transcript-derived) deliberately NOT backed up; documented.
2d. **Approved-term cache for synchronous snapshots**: the store actor publishes an immutable, eagerly loaded approved-terms array (loaded at startup, republished on every decision/restore). Synchronous stop-time paths (`cloudLLMSnapshot`, `makeStopRequest`) snapshot that cache together with `vocabularyText` — no SQLite query on the main actor at stop, temporal guarantee preserved.
2e. **Fit revalidation is continuous, not approval-only**: the combined effective vocabulary is revalidated after every manual `vocabularyText` edit, model change, startup, and backup restore. If manual edits displace previously approved terms past the budget, those terms enter a visible **displaced** state in Settings requiring an explicit eviction decision — the plan never claims a displaced term is active.
3. **Scan execution**: paged projection query (id, transcript, refined only), batched off the main actor, with cancellation and progress; incremental hook after each completed dictation processes just that row. No `allDictations()` full load.
4. **Approval semantics**: approval is gated on the term **provably fitting** the effective vocabulary budget. Conservative fallback bound (tokenizer unloaded): computed over the **exact serialized Whisper prompt bytes** — leading space, every separator, full term list as transcription will render it — with BPE tokens ≥1 byte each, `serialized UTF-8 bytes ≤ token budget` guarantees fit. Tokenizer loaded → exact token count of the same serialized prompt. Does not provably fit → explicit eviction choice or rejection with explanation; approvals never silently inactive; nothing deferred to transcription.
5. **Deterministic ordering** (user-entered first, then approved by `decided_at`) shared by the settings preview, the approval fit-gate, and transcription — one derivation function, one truth.
6. **Settings UI**: ranked suggestions (recurrence desc, cap 25 visible, badge on new), approve/dismiss per term; "Scan library" backfill button with progress + cancel.
7. **Tests**: miner anchored-substitution cases (casing fix, PT accent fix, rewrite → no candidate, translation row → skipped, command-flagged row → skipped), idempotent rescan (counts stable), cascade on dictation delete retracts suggestions but decisions survive Delete All, scan-vs-delete race → benign skip, surface-term tie-break determinism, approval fit-gate over exact serialized prompt (fits/needs-eviction), shared validator parity manual-vs-approved, backup round-trip of approved decisions, all-consumers derivation (CloudLLM snapshot + Apple FM see approved terms).
8. **Observability**: non-content logging of scan progress/failures and skipped-row reasons.

## Key decisions & tradeoffs

- **System-prompt rules + keywords-as-data** (was: template-appended block). Preserves the existing trust boundary; templates stay untrusted; bounded keyword vocabulary prevents prompt injection via Settings.
- **Explicit `VoiceCommandPolicy` per request constructor** (was: assumed single choke point that doesn't exist). Translation and scratchpad transformations are hard-disabled.
- **Sentence-scope grammar with examples** (was: keyword-to-end-of-utterance). Removes the multiple-commands contradiction.
- **Built-in template migration** rather than duplicated instructions; toggle stays meaningful.
- **Default OFF for voice commands, flip out of scope** (round 2 refinement): fallback logs can't measure semantic command failures, so the default-ON flip is a separate future change gated on opt-in feedback + a small EN/PT/provider acceptance suite — not on logging from these PRs.
- **Approved vocabulary in SQLite only, effective vocabulary derived** (was: merge into vocabularyText): kills the impossible cross-store transaction and the provenance ambiguity.
- **SQLite evidence with per-dictation identity** (was: aggregate JSON sidecar). Idempotence and deletion-consistency come from the schema.
- **LLM-context disambiguation over rigid parsing** — unchanged; behavior depends on model quality, accepted with default-off.
- **Suggest + one-click approve, never auto-apply** — unchanged.
- **Automation surface deferred**; builder shaped as the reusable entry point.
- **Separate PRs, A then B.**

## Risks / open questions

- Apple FM on-device model may follow the command rules poorly — default-off + logging covers first release; flip decision needs real usage.
- Template migration touches built-in bodies; existing installs with edited copies of built-ins keep their edits (user-owned), accepted inconsistency documented in release notes.
- Edit-distance heuristic precision for PT diacritics vs genuine rewrites — thresholds (≥2 distinct dictations, ≤2 edit distance) are tunable constants; approval gate is the safety net.
- Byte-bound fit gate is conservative — it may refuse terms that would tokenize fine; acceptable (user can evict), never the inverse failure (silently inactive).

## Out of scope

- Context-aware profiles (exists). Automation endpoints (design-for only). Meeting/system-audio capture. Deterministic command pre-pass. Commands when post-processing disabled. Commands in translation or scratchpad transformation flows.
